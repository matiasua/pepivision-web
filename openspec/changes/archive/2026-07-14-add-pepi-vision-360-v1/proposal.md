## Why

Pepi Visión 360 hoy solo existe como una maqueta de diseño (`design-reference/`): una simulación de una sola página, sin backend, con "persistencia" en `localStorage` del navegador, autenticación de administrador hardcodeada, y ninguna integración real (correo, storage de imágenes, notificaciones). El negocio necesita una primera versión real de la aplicación — sitio público + panel de administración — con base de datos, autenticación real, almacenamiento de imágenes fuera del contenedor y notificaciones reales, manteniendo el mismo contenido, flujos y estilo visual ya validados en el diseño.

**Esta actualización acota el alcance de la implementación actual a un entorno de desarrollo local, 100% reproducible vía Docker Compose.** La infraestructura productiva (instancia, red, TLS/certificados, DNS, base de datos gestionada, backups en la nube, object storage productivo, etc.) **no se crea ni se configura como parte de esta propuesta**: será aprovisionada manualmente por el propietario del proyecto en una etapa posterior, fuera de este alcance. En concreto, esta implementación:
- **No** crea ningún recurso en AWS, ni ejecuta AWS CLI.
- **No** usa herramientas de infraestructura como código (Terraform, CloudFormation, CDK, Pulumi, Ansible ni similares).
- **No** crea instancias Lightsail, buckets, IPs estáticas, registros DNS, certificados TLS ni bases de datos administradas.
- **No** realiza ningún despliegue ni crea pipelines de despliegue hacia AWS.
- **No** usa credenciales de AWS ni LocalStack.

## What Changes

- Empaquetar el entorno de desarrollo completo con **Docker Compose** como **Fase 1 de la implementación, previa a cualquier código de aplicación** (no una fase posterior): servicios `web` (Next.js con hot reload), `postgres`, `migrate` (job puntual de migraciones Prisma), `minio` + `minio-init` (almacenamiento de objetos local y creación automática del bucket), `mailpit` (SMTP local) y `nginx` (reverse proxy local, publicando la app en `http://localhost:8080`, sin HTTPS en desarrollo) — más `adminer`, opcional, tras un profile `tools`. Un único comando `docker compose up --build` debe levantar todo el entorno, construido a partir de `compose.yaml`, `Dockerfile.dev`, `.dockerignore` y `.env.example`.
- **Docker Compose es la base obligatoria de todas las fases siguientes**: ningún comando de la aplicación (`npm`, Prisma, tests, build) se ejecuta directamente en el host — todos corren dentro de los contenedores vía `docker compose exec`/`docker compose run`. El host solo necesita Docker y Docker Compose instalados; no se instala PostgreSQL, MinIO, Nginx ni Mailpit localmente en ningún momento.
- Construir, **una vez que ese entorno de contenedores ya existe**, el **sitio público** de Pepi Visión 360 (Next.js App Router) replicando el contenido y los flujos de `design-reference/`: inicio, catálogo de armazones con filtros/búsqueda, ficha de producto, tipos de cristales, cotizador de lentes (5 pasos), atención a domicilio, nosotros, FAQ, contacto, y las páginas legales (política de privacidad, derechos ARCO, términos y condiciones) — estas últimas marcadas explícitamente como **borrador pendiente de validación legal**.
- Reemplazar toda la "persistencia" en `localStorage` de la maqueta por un backend real: PostgreSQL + Prisma, con modelos para productos, solicitudes (cotizaciones y atención a domicilio), comunas habilitadas, configuración del negocio, usuarios administradores y registro de auditoría.
- Construir un **panel de administración real** con autenticación basada en sesiones, en rutas separadas (`/admin`, `/admin/products`, `/admin/requests`, `/admin/home-visits`, `/admin/settings`, `/admin/users`), reemplazando el login hardcodeado (`admin`/`pepi360`) de la maqueta.
- Introducir **gestión de usuarios y roles administradores** desde el día uno (aunque se cree un único usuario admin inicial), reemplazando el flag booleano `pv360_admin` de la maqueta.
- Introducir **notificaciones reales por correo** (cliente + negocio) cuando se recibe una cotización o una consulta de atención a domicilio, mediante una abstracción SMTP configurable por variables de entorno (`SMTP_*`); en este entorno de desarrollo el correo se envía a **Mailpit** (SMTP local sin autenticación real, con interfaz web para inspeccionar los mensajes) y nunca sale a destinatarios reales. Se mantiene además el CTA "Continuar por WhatsApp" mediante enlaces `wa.me` con mensaje prellenado (sin integrar la WhatsApp Business API en esta versión).
- Introducir **almacenamiento real de imágenes de producto** fuera del contenedor de la aplicación, mediante una abstracción de almacenamiento de objetos configurable por variables de entorno (`OBJECT_STORAGE_*`) y compatible con S3; en este entorno de desarrollo corre localmente contra **MinIO**, reemplazando el patrón actual de fotos codificadas en base64 dentro de `localStorage`. El proveedor de object storage productivo se decide y configura cuando se aprovisione la infraestructura productiva (fuera de este alcance).
- Introducir una **lista configurable de comunas habilitadas** para atención a domicilio, administrable desde `/admin/home-visits`, reemplazando el campo de comuna en texto libre sin validar de la maqueta.
- Definir una **política de retención de solicitudes** (12 meses por defecto, configurable) y documentar el mecanismo futuro de eliminación/anonimización (no se implementa el job automático en esta versión, pero el modelo de datos lo soporta).
- Reemplazar el formulario de derechos ARCO de la maqueta (que hoy solo muestra una confirmación en pantalla sin persistir nada) por una **capacidad real de gestión de solicitudes de derechos ARCO** (`data-rights-requests`): persistencia en PostgreSQL, notificación por correo al negocio, visibilidad en el panel administrativo, flujo de estados (`RECEIVED` → `IN_REVIEW` → `RESOLVED`/`REJECTED`) con notas de resolución, y registro de cada cambio de estado en el log de auditoría.
- **No** se incorpora carrito, checkout ni pago en línea — las ventas y cotizaciones se siguen cerrando por WhatsApp, tal como en la maqueta.
- **Actualización aprobada durante la implementación**: el cotizador permite, de forma opcional, adjuntar el archivo de la receta óptica (PDF/JPG/PNG/WebP) junto con la indicación de si el cliente posee receta. El archivo se almacena en un bucket privado separado del de fotografías de producto (`PRIVATE_OBJECT_STORAGE_BUCKET`), sin acceso anónimo y sin URL pública permanente — solo accesible desde el panel administrativo mediante una URL firmada de corta duración, generada tras verificar sesión y autorización, y cada acceso queda registrado en el log de auditoría. Ver `specs/quote-requests/spec.md` y `specs/request-inbox/spec.md` para el detalle normativo, y `design.md` → "Almacenamiento privado de adjuntos de solicitudes" para la arquitectura. Tampoco se adjuntan ni almacenan documentos sensibles en las solicitudes de derechos ARCO — ese punto no cambió.
- **Actualización aprobada durante la implementación**: introducir la entidad `Brand` (marca comercial del armazón), relacionada con `Product` mediante `brandId` (nullable, para no invalidar productos existentes creados antes de esta adición), poblada mediante un seed idempotente a partir de los logos reales en `public/marcas/`, con selector en el formulario administrativo de modelos y con visualización/filtro por marca en el catálogo público. **`Brand` en esta v1 es una capacidad concreta e inicial** — no es todavía una arquitectura de catálogo multi-categoría; ese diseño extensible (categorías administrables, ofertas comerciales por categoría, atributos dinámicos) se aborda como una propuesta separada en `redesign-extensible-catalog-v2`, sin que esta propuesta la anticipe ni la sustituya.
- **Actualización aprobada durante la implementación**: reemplazar el modelo de galería de fotografías de 3 posiciones fijas (`ImageSlot`: principal/frontal/lateral) por una galería de largo variable donde cada fotografía se asocia a uno de los colores del producto (`ProductColor`), con orden explícito (`sortOrder`), una portada designable (`isCover`), y soporte para reemplazar, eliminar y reordenar. Agregar/quitar un color pasa a ser una mutación inmediata e independiente del guardado general del formulario (ya no requiere guardar el modelo completo y volver a entrar para que el color quede disponible para asignarle fotografías).
- **Actualización aprobada durante la implementación**: permitir iniciar sesión en `/admin` con nombre de usuario (`username`, además del correo electrónico ya previsto), con el mismo mensaje de error genérico independientemente de cuál de los dos identificadores se usó.
- **Actualización aprobada durante la implementación**: los correos transaccionales (confirmación de cotización, notificación de cotización al negocio, confirmación de atención a domicilio, notificación de atención a domicilio, notificación de derechos ARCO) se generan como HTML profesional (tabla, estilos inline, compatible con clientes de correo reales) acompañado siempre de una versión en texto plano equivalente — nunca solo uno de los dos. El HTML incluye el logo mediante una URL absoluta configurable (`EMAIL_ASSET_BASE_URL`, con `APP_URL` como respaldo), escapa todo contenido dinámico proveniente de un formulario público, y nunca incluye el archivo de una receta adjunta, su clave de almacenamiento (`storageKey`) ni una URL privada — solo un aviso de que existe, disponible en el panel administrativo.
- Alojar las tipografías (Poppins, Inter) dentro de la propia aplicación, eliminando la dependencia de Google Fonts CDN presente en la maqueta.
- Mantener el dominio/URL de la aplicación configurable mediante las variables de entorno `APP_URL` y `NEXT_PUBLIC_APP_URL`, sin acoplar el código a un dominio o proveedor DNS específico; en este entorno de desarrollo apuntan a `http://localhost:8080`.
- Fijar el objetivo de accesibilidad en WCAG 2.1 nivel AA, verificado mediante comprobaciones automáticas (axe y Lighthouse CI) más una lista de validación manual (teclado, foco, formularios, contraste, textos alternativos, lector de pantalla); no se exige una auditoría de accesibilidad externa para el lanzamiento inicial.
- El empaquetado, despliegue y operación de un entorno **productivo** (Docker Compose + Nginx con HTTPS, CI/CD hacia infraestructura real, backups automatizados, ambientes de staging, aprobación manual de despliegue, etc.) queda **fuera del alcance de esta propuesta**: se retoma como una propuesta separada cuando el propietario del proyecto haya aprovisionado manualmente la infraestructura productiva.

**BREAKING**: no aplica — no existe ninguna versión productiva previa que romper; esta propuesta reemplaza por completo el prototipo de `design-reference/`, que permanece intacto como referencia de solo lectura y no como código a migrar.

## Capabilities

### New Capabilities

- `public-site`: páginas públicas informativas del sitio (inicio, tipos de cristales, nosotros, FAQ, contacto, y las páginas legales de privacidad/ARCO/términos), navegación global (header, footer, banner de cookies, botón flotante de WhatsApp) y configuración de negocio que alimenta esos contenidos.
- `product-catalog`: catálogo público de armazones (listado, búsqueda, filtros por género/forma/material/color/marca/precio/disponibilidad), ficha de detalle de producto con galería de fotografías por color y marca visible, incluyendo productos relacionados.
- `product-management`: gestión administrativa del catálogo (alta/edición/baja de modelos, marca, colores como mutación inmediata, galería de fotografías por color, disponibilidad, etiquetas) desde `/admin/products`.
- `quote-requests`: flujo público del cotizador de lentes (5 pasos: armazón —con marca y color resueltos desde PostgreSQL—, cristal, tratamientos, indicación de receta con adjunto opcional y privado del archivo, datos de contacto), envío de la solicitud, notificación por correo HTML+texto plano al cliente y al negocio, y CTA de continuidad por WhatsApp.
- `home-visit-requests`: flujo público de consulta de atención a domicilio (validado contra la lista de comunas habilitadas), envío de la solicitud, notificación por correo HTML+texto plano al cliente y al negocio, y CTA de WhatsApp.
- `request-inbox`: bandeja administrativa unificada de solicitudes (cotizaciones + atención a domicilio) en `/admin/requests`, con filtro por tipo, cambio de estado (nueva/atendida), acceso autenticado a la receta adjunta mediante URL firmada temporal, eliminación, y aplicación de la política de retención configurable.
- `home-visit-coverage`: gestión administrativa de la lista de comunas habilitadas para atención a domicilio, en `/admin/home-visits`.
- `business-settings`: configuración administrativa de los datos del negocio (WhatsApp, teléfono visible, correo, Instagram, horario, ubicación, período de retención de solicitudes) en `/admin/settings`.
- `admin-auth`: autenticación basada en sesiones para el panel de administración (inicio de sesión con correo o con nombre de usuario), con modelo de usuarios y roles administradores (soporte multiusuario desde el diseño, aunque v1 solo crea un usuario), gestión de usuarios en `/admin/users`, y registro de auditoría de acciones administrativas.
- `product-image-storage`: subida, validación, almacenamiento y entrega de imágenes de producto en una galería de largo variable asociada a color, usando una abstracción de almacenamiento de objetos externo compatible con S3 (en desarrollo, MinIO) y un bucket separado y privado (sin acceso anónimo) para adjuntos sensibles de solicitudes, fuera del contenedor de la aplicación.
- `data-rights-requests`: gestión real de solicitudes de derechos ARCO — persistencia en PostgreSQL, notificación por correo al negocio, visibilidad y flujo de estados en el panel administrativo, y registro de auditoría por cada cambio de estado.

### Modified Capabilities

_Ninguna — no existen specs previas en `openspec/specs/`; este es el primer conjunto de capacidades del proyecto._

## Impact

- **Código nuevo**: aplicación Next.js completa (App Router) — no existe código de aplicación previo, solo la maqueta de referencia en `design-reference/` (no se modifica) y este árbol de documentación/planificación.
- **Base de datos**: nuevo esquema PostgreSQL (vía Prisma) para productos, solicitudes comerciales, solicitudes de derechos ARCO, comunas habilitadas, configuración de negocio, usuarios administradores, sesiones y auditoría.
- **Infraestructura**: exclusivamente local — Docker Compose con los servicios `web`, `postgres`, `migrate`, `minio`, `minio-init`, `mailpit`, `nginx` (más `adminer` opcional vía profile `tools`), todos en una red interna de Compose, con volúmenes persistentes para `postgres` y `minio`, construida en la **Fase 1** de `tasks.md` (antes que cualquier código de aplicación) y usada como base de ejecución de todas las fases siguientes. **No se crea ni configura ningún recurso en AWS** (ni instancias, ni buckets, ni IPs estáticas, ni DNS, ni certificados, ni bases de datos administradas), no se ejecuta AWS CLI, no se usa Terraform/CloudFormation/CDK/Pulumi/Ansible ni LocalStack, y no se realiza ningún despliegue. El dominio/URL de la app es configurable vía `APP_URL`/`NEXT_PUBLIC_APP_URL` (en este entorno, `http://localhost:8080`).
- **Flujo de trabajo**: todo comando de la aplicación (`npm`, Prisma, tests, build) se ejecuta dentro de los contenedores vía `docker compose exec`/`docker compose run`; el host de cada desarrollador solo necesita Docker y Docker Compose, sin instalaciones locales de PostgreSQL, MinIO, Nginx ni Mailpit.
- **Dependencias externas**: ninguna en este entorno — MinIO y Mailpit corren localmente dentro del propio Docker Compose; WhatsApp solo como enlace `wa.me` (sin API). El proveedor de correo y de almacenamiento de objetos productivos quedan pendientes de decidir cuando se aprovisione la infraestructura productiva (fuera de este alcance).
- **Contenido legal**: los textos de privacidad/ARCO/términos de la maqueta se reutilizan como borrador, marcados en la propia UI y en `design.md` como pendientes de validación legal antes de un lanzamiento real con usuarios finales.
- **Accesibilidad**: se incorpora axe y Lighthouse CI como comprobaciones automáticas en CI, más una lista de validación manual documentada en `design.md`; no se exige auditoría externa para este hito.
- **Fuera de alcance de esta propuesta**: WhatsApp Business API, carrito/checkout/pago en línea, subida de documentos adjuntos en solicitudes de derechos ARCO, automatización real de borrado/anonimización por retención (solo se documenta el mecanismo futuro), auditoría de accesibilidad externa, la arquitectura extensible de catálogo multi-categoría (categorías administrables, ofertas comerciales por categoría — ver `redesign-extensible-catalog-v2`), **toda creación o configuración de infraestructura productiva (AWS/Lightsail u otra), cualquier despliegue, y cualquier pipeline de despliegue** — la infraestructura productiva será creada manualmente por el propietario del proyecto en una etapa posterior, como trabajo separado de esta propuesta. (El adjunto opcional y privado de receta óptica en el cotizador **ya no está fuera de alcance** — ver "What Changes".)
